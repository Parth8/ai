fase zero

The Mental Model

Understand the landscape before touching a line of code. What an LLM is, what it isn't, and why this wave is different from every AI hype cycle before it.

2–3 days · ~5 hours of material

When this is done: you can explain what a language model actually does — next-token prediction at extraordinary scale — and describe the modern AI stack (foundation model → prompting → RAG → agents → product) to anyone.

Concepts to own

What an LLM actually is
A statistical next-token predictor trained on most of human text. Not a brain, not a database, not a search engine. Pattern completion at a scale where useful reasoning-like behaviour emerges.
The model landscape
Frontier closed models (GPT, Claude, Gemini) versus open-weight models (Llama, Mistral, Qwen, DeepSeek). Who builds them, how they're priced, and what the tradeoffs are for someone building a product.
The AI stack
Foundation model → fine-tuning → prompting → retrieval (RAG) → agents → product. Each layer adds capability without retraining the base model. Almost every AI product is a composition of these layers.
Why now is different
Scaling laws, instruction tuning, and reinforcement learning from human feedback turned raw text completers into steerable, general-purpose engines. Tool use then turned them into things that can act.
The core vocabulary
Token, context window, parameters, inference, training, embedding, hallucination, temperature. Own these eight words before anything else — every other concept builds on them.

fase uno

How LLMs Work

The internals: transformers, attention, tokens, context, training. You don't need to build a model professionally — you need to understand one deeply enough to teach it.

1 week · ~12 hours of material

When this is done: you can explain transformers, attention, tokenization, context windows, and the full training pipeline (pretraining → instruction tuning → RLHF) to a CEO or an engineer, at whichever depth they need.

Concepts to own

Tokenization
Text becomes tokens via byte-pair encoding. Roughly ¾ of a word per token in English, far worse in other languages. Tokenization explains cost, context limits, and many strange model failures (spelling, arithmetic, rare words).
Transformers & attention
The 2017 architecture behind everything. Self-attention lets every token weigh its relationship to every other token in parallel — that's the mechanism that made scale possible and killed recurrent networks.
Pretraining
Next-token prediction over trillions of tokens. This is where world knowledge gets compressed into weights. Compute plus data equals capability, with a predictability described by scaling laws.
Instruction tuning & RLHF
A pretrained model is a wild autocomplete. Supervised fine-tuning on instruction data makes it helpful; reinforcement learning from human feedback makes it steerable and safe. This combination was the ChatGPT breakthrough.
Context windows
The model's working memory: everything inside is visible, nothing outside exists. Modern windows range from 8K to over a million tokens, but retrieval quality degrades with position — "lost in the middle" is real.
Parameters & scale
7B, 70B, 400B+ — parameter count roughly sets the capability ceiling, but training data quality and post-training now matter as much. Small models are catching up fast for narrow tasks.
Sampling: temperature, top-p
The model outputs a probability distribution over the next token; sampling settings decide how it picks. Temperature 0 for extraction and evals, ~0.7–1.0 for creative work. Knowing which knob to turn is a practitioner skill.

fase due

Prompting & APIs

Your primary interface with every model, forever. Most people never actually master this — which is exactly why it's a differentiator.

1–2 weeks · ~8 hours of material plus practice

When this is done: you can reliably extract structured, predictable output from any LLM using system prompts, few-shot examples, and chain-of-thought — and you can call any model API with confidence.

Concepts to own

System prompts
The instruction frame that governs model behaviour: persona, constraints, output format, tone, refusal rules. Most AI products live or die on the quality of this one document.
Zero-shot vs few-shot
Zero-shot: just ask. Few-shot: show examples first. Few-shot is dramatically more reliable for structured or stylistically-precise output, at the cost of tokens.
Chain-of-thought
Asking the model to reason step by step measurably reduces errors on complex tasks. Reasoning models (o-series, R1 and successors) now bake this in with dedicated "thinking" tokens.
Structured output & function calling
JSON mode, schemas, and function calling force the model to return parseable data instead of prose. This is the foundation of every product integration — and of agents, in the next phase.
Context engineering
The term replacing "prompt engineering": deciding what goes into the context window, in what order, and why. Instructions, examples, retrieved documents, tool results, history — composition is the skill.
API mechanics
Messages arrays (system / user / assistant turns), max tokens, stop sequences, streaming, rate limits, prompt caching, and cost per token. Know these before building anything.

Embeddings, Vectors & RAG

How AI systems gain knowledge beyond their training data. Retrieval-augmented generation is still the dominant production pattern for knowledge-intensive products.

1–2 weeks · ~10 hours of material plus one build

When this is done: you can design and build a working RAG pipeline end-to-end, explain why each stage exists, diagnose where retrieval quality breaks, and evaluate the whole thing with real metrics.

Concepts to own

Embeddings
Dense vectors that represent meaning. "Cat" and "feline" land close together in embedding space; "cat" and "democracy" don't. This is how machines do semantic similarity — the foundation of search, recommendations, clustering, and RAG.
Vector databases
Databases optimised for approximate nearest-neighbour search in high-dimensional space: Pinecone, Weaviate, Qdrant, Chroma, pgvector, and vector features inside Postgres and every major cloud DB. Know two well; know when plain Postgres with pgvector is enough (often).
The RAG pipeline
Ingest → chunk → embed → store → (query → embed → retrieve → rerank) → inject into context → generate with citations. Every stage has failure modes; being able to name them is what separates practitioners from tourists.
Chunking strategies
How you split documents affects retrieval quality more than almost anything else. Fixed-size, recursive, semantic, and document-structure-aware chunking each have a use case. Bad chunking cannot be fixed downstream.
Hybrid search & reranking
Dense (semantic) plus sparse (keyword/BM25) search almost always beats either alone. A cross-encoder reranker on top of retrieval is the cheapest big precision win available.
Evaluating RAG
Faithfulness, answer relevancy, context precision, context recall — the RAGAS framework's four dials. If you can't measure retrieval quality, you can't improve it.
When not to RAG
Long context windows, prompt caching, and agentic search have eaten some of RAG's territory. Knowing when to just stuff documents into context — or let an agent search — is a current, live product judgment.

Agents, Loops & Tools

The single most important idea in applied AI right now: a model, given tools and a loop, becomes a system that can act. Everything currently being built lives here.

2 weeks · ~10 hours of material plus one build

When this is done: you can explain exactly what an agent is (LLM + tools + loop), trace an agent's reasoning cycle step by step, name the major failure modes, and build a simple tool-using agent without a framework.

Concepts to own

The agent loop
Reason → choose a tool → act → observe the result → reason again, until done or stuck. That loop, popularised by the ReAct paper, is the whole trick. Every agent framework is scaffolding around this cycle.
Tool use / function calling
The model emits a structured call (search the web, query a database, run code, send an email); the host executes it and feeds the result back into context. Tools are how a text predictor gets hands.
Workflows vs agents
A workflow is a predefined chain of LLM calls; an agent decides its own path at runtime. Workflows are cheaper, more reliable, and right for most products. Knowing when you actually need agency is a core product judgment.
Agent memory
Four kinds: in-context (the conversation), external (a vector store), episodic (summaries of past sessions), and procedural (learned routines). Long-running agents live or die on memory design.
Planning & decomposition
Good agents break big goals into steps, track progress, and recover from failures. Techniques: explicit plan-then-execute, todo lists in context, reflection steps, and human checkpoints at irreversible actions.
Agent failure modes
Infinite loops, hallucinated tool parameters, goal drift, context overflow, cascading errors, and prompt injection through tool results. Reliability engineering here is a product discipline, not an afterthought.
Agentic RAG
Instead of one-shot retrieval, the agent decides when to search, reformulates queries, evaluates results, and searches again. This is rapidly replacing static RAG for complex questions.

MCP & Multi-Agent Systems

The infrastructure layer of the agentic era: standard protocols for connecting agents to tools, and patterns for making multiple agents work together.

1–2 weeks · ~8 hours of material

When this is done: you can explain MCP and A2A and why standard protocols matter, build or spec an MCP server, and reason about multi-agent architectures — orchestrator-worker, sequential, hierarchical — like a systems designer.

Concepts to own

MCP — Model Context Protocol
An open standard (originated by Anthropic, now industry-wide) for connecting AI applications to tools and data sources. Think USB-C for AI: any client, any server, one protocol. Its primitives — tools, resources, prompts — are worth knowing cold.
Why protocols beat integrations
Before MCP: N models × M tools = N×M custom integrations. After: N + M. This is the same story as HTTP, SQL, and REST — and understanding it lets you predict where the ecosystem is heading.
A2A — Agent-to-Agent protocol
Google-originated, Linux Foundation-governed standard for agents discovering and talking to each other. MCP connects agents to tools; A2A connects agents to agents. Together they're the emerging agentic stack.
Multi-agent patterns
Sequential (pipeline of specialists), orchestrator-worker (a lead agent delegates to subagents), hierarchical (managers of managers), and debate/critique loops. More agents means more capability and much more failure surface — most products need fewer agents than founders think.
Orchestration frameworks
LangGraph (stateful graphs, production-leaning), CrewAI (role-based teams, fast to prototype), AutoGen/Agent Framework (Microsoft, conversation-driven), smolagents (Hugging Face, minimal, code-first). Learn one deeply — LangGraph is the safest deep bet — and stay conversant in the rest.
Agent observability
Tracing every step — which tool was called, with what input, what came back, what it cost — via LangSmith, Langfuse, or similar. Debugging a multi-agent system without traces is archaeology.

AI Product Craft

Where engineering stops and product thinking starts: evals, reliability, cost, failure modes, and the judgment calls that make AI features shippable.

2–4 weeks · ongoing practice

When this is done: you can spec, evaluate, and ship AI products. You know how to measure quality in a probabilistic system, manage cost and latency, design for failure, and speak fluently to engineers and executives alike.

Concepts to own

Evals — the number one skill
LLM output is probabilistic, so quality must be measured systematically: golden datasets, LLM-as-judge, human review loops, regression suites on every prompt change. "It feels good" is not an eval. Teams that master evals ship faster and safer than teams that don't — this is the highest-leverage skill in AI product work.
The AI product patterns
Chatbot, copilot, autonomous agent, background workflow, and multimodal app. Every AI product is one of these or a combination — and each has distinct UX, trust, and reliability requirements.
Hallucination as a product problem
Grounding, citations, constrained output, confidence signals, and human-in-the-loop are product strategies, not just model fixes. The right answer depends on the cost of being wrong in your domain.
Cost & latency economics
Model tiers differ in price by 20–100×. Prompt caching cuts repeat-prefix costs dramatically; batching, streaming, and routing cheap requests to small models are PM-level decisions that make or break unit economics.
Observability
Trace every call: inputs, outputs, latency, cost, tool use. LangSmith, Langfuse, Helicone. Non-negotiable in production — you cannot improve what you cannot see.
Safety & prompt injection
Any text your system ingests — documents, web pages, tool results — can contain instructions that hijack the model. Injection is unsolved; defense is layered: input handling, least-privilege tools, human checkpoints on consequential actions.
Build vs buy vs prompt vs fine-tune
The canonical decision ladder: try prompting first, then RAG, then fine-tuning, then custom models — escalating only when the previous rung measurably fails. Most production problems die on rung one or two.

Build & Ship End-to-End

Learning compounds when it's deployed. Three real projects, on real URLs, with real data — this is where everything before it becomes yours.

1–2 months · project-driven

When this is done: you have two or three deployed AI products in your portfolio and can architect the full stack — ingestion → retrieval → agent → API → UI → observability — from a blank page.

Concepts to own

Full-stack AI architecture
A UI layer (Gradio, Streamlit, or Next.js) → an API layer (FastAPI) → orchestration (LangGraph or plain Python) → model APIs → vector plus relational storage → tracing (Langfuse). Be able to sketch this on a whiteboard from memory.
Local models with Ollama
Run Llama, Mistral, Qwen and friends on your own machine: zero cost, full privacy, instant iteration. Essential for prototyping and for understanding what open models can and cannot do.
The open-source ecosystem
Hugging Face Hub hosts hundreds of thousands of models, datasets, and demos. Loading and running any model in ten lines of Python is a baseline capability, not an advanced one.
Deployment
Hugging Face Spaces for demos; Railway, Render, or Vercel for apps; Modal or Replicate for model serving; Docker for packaging. Shipping something ugly to a real URL in week one beats polishing localhost for a month.
The three projects that matter
One: a RAG system over a real document corpus you care about, with RAGAS evals. Two: an agent that uses three or more tools to complete a multi-step task, with traces. Three: an MCP server exposing a data source you know well. Ship all three, write up each one.

Fine-tuning & Staying Sharp

The last technical layer — when prompting and RAG aren't enough — plus the permanent learning system, because this field reinvents itself every few months.

1–2 months, then forever

When this is done: you know when fine-tuning is justified and how it works (LoRA, QLoRA, preference optimisation), you can evaluate any new model or tool within thirty minutes, and you have a weekly system for tracking the frontier.

Concepts to own

When to fine-tune
When you need a style or format prompting can't reliably produce, have at least ~1,000 high-quality examples, or need a small cheap model to match a big one on a narrow task. It's the last resort on the decision ladder, not the first move.
LoRA & QLoRA
Parameter-efficient fine-tuning: train a small adapter instead of all the weights. QLoRA adds quantisation so large models can be tuned on a single GPU. This is how almost all practical fine-tuning happens.
Preference optimisation & RL
RLHF, DPO, and GRPO — how models are aligned to human preferences and trained to reason. Understanding these at a conceptual level lets you read any model release announcement critically.
Multimodal & beyond text
Vision-language models, speech models, image and video generation, and computer-use agents. The product frontier is multimodal; keep a working mental map even if you build with text first.
Benchmark literacy
MMLU, GPQA, SWE-bench, LMArena — know what each measures, who runs it, and why to be sceptical of every vendor-reported number. Independent leaderboards over press releases, always.
The 30-minute model eval
New model drops: read the system card, check independent benchmarks, run your own hardest prompts, check pricing, read two practitioner takes. You can have a defensible opinion in half an hour — this is the skill that keeps you current forever.

Glossario · Pin this
Token
The unit of text a model reads and writes — roughly ¾ of an English word.
Context window
The model's working memory. Inside is visible; outside doesn't exist.
Embedding
A vector of numbers representing meaning; similar meanings sit close together.
Vector database
A store optimised for finding the nearest vectors — i.e. searching by meaning.
RAG
Retrieve relevant documents at query time and hand them to the model, instead of retraining it.
Agent
A model with tools and a loop: reason, act, observe, repeat until done.
Tool / function calling
The model emits a structured request; your code executes it and returns the result.
MCP
Open protocol connecting AI apps to tools and data. USB-C for AI.
A2A
Open protocol for agents to discover and communicate with other agents.
Fine-tuning
Continuing a model's training on your own examples to change its behaviour.
Transformer
The neural architecture behind every modern LLM, built on attention.
Attention
The mechanism letting each token weigh its relationship to every other token.
Hallucination
Fluent, confident output that is simply wrong. A product problem, not just a model one.
Eval
A systematic measure of output quality. The discipline separating serious AI teams from the rest.
